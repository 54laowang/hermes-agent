# 记忆管理 SOP

## 触发条件
- 需要写入新记忆
- 整理会话发现重要信息
- 用户明确要求"记住这个"

## 信息分类决策树

```
收到新信息
    ↓
能指导具体行动吗？
    ├─ 否 → 归档到 L4，不写入上层
    └─ 是 ↓
          是事实还是推测？
              ├─ 推测 → 归档到 L4 + 标注"推测"
              └─ 事实 ↓
                    使用频率？
                        ├─ 极高 → L1（红线规则/高频指针）
                        ├─ 中等 → L2（全局事实库）
                        └─ 重复场景 → 创建 L3 SOP
```

## 写入检查清单

写入前必须回答：
- [ ] 这条信息能指导什么具体行动？
- [ ] 这是验证过的事实吗？来源是什么？
- [ ] 放在哪一层最合适？为什么？
- [ ] 如果放 L1，是否符合 ≤30 行约束？
- [ ] 是否需要创建配套的 SOP？

## L1 写入规则

**红线规则**（绝对不可违背）：
- 格式：`场景 → 强制动作`
- 触发：每次相关场景必须执行
- 示例：`市场分析 → 先获取datetime，复核数据日期`

**高频指针**（快速定位）：
- 格式：`场景 → 文件路径/具体行动`
- 目标：3 秒内找到相关信息
- 示例：`财经分析 → L2市场规则 + market_analysis_sop`

## L2 写入规则

**全局事实标准格式**：
```
## 分类名称
- 键：值
- 键：值（括号补充说明）

示例：
## 微信推送配置
- 推送目标：o9cq80znDxnUb_ojFoc-8kBCdqdE@im.wechat
- 已废弃字段：deliver (cronjob异步坑)
- 正确方式：任务内send_message
```

**验证标记**：
- 每条事实必须包含来源或验证时间
- 底部添加：`最后验证：YYYY-MM-DD`

## L3 SOP 创建规则

**标准结构**：
```markdown
# 任务名称 SOP

## 触发条件
- 用户说 "..."
- 遇到 ... 情况

## 执行流程
1. 步骤一
   - 操作：...
   - 关注点：...
   - 常见坑：...
2. 步骤二
   ...

## 质量检查
- [ ] 检查项 1
- [ ] 检查项 2

## 坑点提示
- 坑1 → 解决方案
- 坑2 → 解决方案
```

## 同步规则

**冲突处理**：
1. L1 > L2 > L3（上层优先）
2. 新信息覆盖旧信息时，必须在 L4 保留历史版本
3. 发现矛盾时，暂停并提示用户确认

**更新流程**：
1. 修改 L1/L2 时，先备份到 L4
2. 更新后验证：相关任务执行 3 次无异常
3. 记录更新时间和原因

## 坑点提示

**坑1：过度写入 L1**
- 症状：L1 超过 30 行，查找变慢
- 解决：定期清理，仅保留红线规则和真正高频指针

**坑2：L2 包含推测**
- 症状：基于 L2 决策出错
- 解决：严格区分事实与推测，推测归档到 L4 并标注

**坑3：SOP 过于复杂**
- 症状：执行 SOP 比不执行还慢
- 解决：只保留关键步骤，移除"正确但无用"的细节

**坑4：忘记验证**
- 症状：记忆过期导致错误决策
- 解决：L2 每条事实必须有验证时间戳
