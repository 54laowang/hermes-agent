# Plan Mode SOP (四阶段流程)

## 触发条件

满足以下任一条件即触发 Plan Mode：
- 任务涉及 3 个以上步骤
- 需要外部信息搜集
- 执行过程可能出错
- 用户明确要求详细规划
- 涉及重要决策或不可逆操作

## 四阶段执行流程

### Phase 1: 探索 (Explore)

**目标**：充分理解任务，收集必要信息

**步骤**：
1. 分析用户意图
   - 用户真正想要什么？
   - 有没有隐含的目标或约束？
   - 成功的标准是什么？
   
2. 搜索相关信息
   - 项目文档：README、CLAUDE.md、相关代码
   - 知识库：L1/L2/L3 是否有相关记忆
   - 外部资源：如果需要，进行网页搜索
   
3. 识别风险
   - 可能出错的环节
   - 依赖的外部因素
   - 不可逆的操作点

**输出**：
```
## 任务理解
用户意图：...
隐含目标：...
成功标准：...

## 信息收集
项目上下文：...
知识库记忆：...
外部资源：...

## 风险识别
- 风险1：...
- 风险2：...
```

**时间预算**：不超过总任务时间的 20%

### Phase 2: 规划 (Plan)

**目标**：制定清晰、可执行的计划

**步骤**：
1. 任务分解
   - 将大任务拆解为子任务
   - 每个子任务应该是原子性的
   - 明确子任务之间的依赖关系
   
2. 确定执行顺序
   - 拓扑排序依赖关系
   - 标识可并行执行的子任务
   - 规划检查点
   
3. 资源分配
   - 需要哪些工具/技能
   - 每个步骤的预期输出
   - 错误处理策略

**输出**：
```markdown
## 执行计划

### Task 1: [子任务名称]
- 目标：...
- 工具：...
- 预期输出：...
- 注意事项：...

### Task 2: [子任务名称]
- 依赖：Task 1
- ...

## 检查点
- Checkpoint 1: 完成 Task 1 后，验证...
- Checkpoint 2: ...
```

**时间预算**：不超过总任务时间的 15%

### Phase 3: 执行 (Execute)

**目标**：按计划高质量完成各子任务

**步骤**：
1. 按顺序执行子任务
2. 记录关键决策和原因
3. 遇到异常时：
   - 评估影响范围
   - 决定是否需要重新规划
   - 如果需要，回到 Phase 2
   
4. 在检查点验证中间结果

**执行原则**：
- 严格按计划执行，避免范围蔓延
- 每完成一个子任务立即验证
- 遇到问题先尝试解决，失败则回退

**输出追踪**：
```
## 执行日志

### [HH:MM] Task 1 开始
- 操作：...
- 结果：✅ 成功 / ❌ 失败
- 关键决策：...

### [HH:MM] Task 2 开始
- ...
```

**时间预算**：占总任务时间的 50-60%

### Phase 4: 验证 (Verify)

**目标**：确保成果质量，发现潜在问题

**步骤**：
1. 功能验证
   - 是否达成用户的原始目标？
   - 所有子任务都完成了吗？
   - 边界情况处理是否完善？
   
2. 对抗性检查（核心！）
   - 如果我是最挑剔的用户，我会怎么挑毛病？
   - 有没有更简单高效的方案？
   - 如果用户环境不同，这个方案还能工作吗？
   - 长期来看，这个方案可维护吗？
   
3. 整理输出
   - 清理临时文件、调试代码
   - 整理文档和注释
   - 总结经验教训

**对抗性检查问题清单**：
- [ ] 用户拿到这个结果，能直接使用吗？
- [ ] 有没有遗漏边界情况？
- [ ] 错误处理是否完善？
- [ ] 性能是否可接受？
- [ ] 代码/方案是否易于理解和维护？
- [ ] 如果由其他人接手，能看懂吗？
- [ ] 有没有过度设计？
- [ ] 有没有更简单的替代方案？

**输出**：
```markdown
## 验证报告

### 功能验证
- ✅ 原始目标已达成
- ✅ 所有子任务已完成
- ⚠️ 边界情况：...

### 对抗性检查
- 发现问题1 → 已修复
- 发现问题2 → 建议优化（非阻塞）

### 最终成果
[完整的成果交付]

### 经验总结
- 本次踩坑：...
- 可复用经验：... → 考虑创建 SOP
```

**时间预算**：占总任务时间的 15-20%

## 异常处理

### 需要重新规划的情况
- 发现新的重要信息，影响原有计划
- 多个子任务失败，无法继续
- 用户需求发生重大变化

### 回退策略
- Phase 3 遇到阻塞 → 回退到 Phase 2 重新规划
- Phase 2 发现信息不足 → 回退到 Phase 1 补充调研
- Phase 4 发现重大缺陷 → 回退到 Phase 3 修复

## 最佳实践

1. **探索充分比急于执行重要**
   - 宁可多花时间理解需求，也不要走弯路
   
2. **计划要具体到原子级别**
   - 模糊的计划等于没有计划
   
3. **执行时保持专注**
   - 一次只做一个子任务
   - 不被"顺便可以..."的想法带偏
   
4. **验证必须包含对抗性检查**
   - 自己满意不算，要经得起最挑剔的眼光

## 坑点提示

**坑1：跳过探索直接规划**
- 症状：执行到一半发现理解错了需求
- 解决：Phase 1 必须确认 3 个问题：用户意图、信息完整、风险识别

**坑2：计划过于宏大**
- 症状：计划包含几十个子任务，无法执行
- 解决：子任务控制在 7±2 个，超过就再分层

**坑3：执行时范围蔓延**
- 症状：执行过程中不断加新功能
- 解决：严格按计划执行，新想法记录到"后续优化"列表

**坑4：验证走过场**
- 症状：快速扫一眼就说"没问题"
- 解决：对抗性检查必须至少发现 1 个潜在问题
