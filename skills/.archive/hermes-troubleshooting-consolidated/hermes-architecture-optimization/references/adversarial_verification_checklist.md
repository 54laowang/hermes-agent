# 对抗性验证快速检查卡片

> 核心理念：完成后站在对立面挑毛病，避免低级错误

## 使用时机

- ✅ 重要任务完成前
- ✅ 交付用户成果前
- ✅ 代码合并到主分支前
- ✅ 部署到生产环境前
- ✅ 用户验收前

## 快速检查清单（5 分钟版）

### 1. 用户视角（必问）

- [ ] 用户拿到这个结果，能直接使用吗？
- [ ] 用户的环境/系统/版本不同时，还能工作吗？
- [ ] 用户能理解如何使用吗？需要额外的文档/说明吗？
- [ ] 如果用户是新手，能看懂吗？

### 2. 边界情况（必问）

- [ ] 输入为空/空字符串/null 时会怎样？
- [ ] 输入异常大/长时会怎样？
- [ ] 并发/同时多个请求时会怎样？
- [ ] 依赖的服务不可用时会怎样？
- [ ] 用户取消/中断操作时会怎样？

### 3. 错误处理（必问）

- [ ] 所有可能的失败点都有错误处理吗？
- [ ] 错误信息对用户友好吗？还是技术黑话？
- [ ] 错误发生时，用户能知道下一步该怎么做吗？
- [ ] 有日志记录方便排查吗？

### 4. 性能与资源（必问）

- [ ] 响应时间可接受吗？（用户愿意等多久？）
- [ ] 内存/CPU/网络占用合理吗？
- [ ] 如果数据量增加 10 倍，还能工作吗？
- [ ] 有没有明显的性能瓶颈？

### 5. 可维护性（必问）

- [ ] 代码/方案结构清晰吗？
- [ ] 变量/函数命名能一眼看懂吗？
- [ ] 关键逻辑有注释吗？
- [ ] 如果由其他人接手，能看懂吗？
- [ ] 修改一个功能需要改很多地方吗？（耦合度）

### 6. 安全性（必问）

- [ ] 用户输入有验证/过滤吗？
- [ ] 有没有 SQL 注入/XSS/命令注入风险？
- [ ] 敏感信息（密码/密钥）有保护吗？
- [ ] 权限检查完善吗？

### 7. 对立面试问（核心！）

**假设你是最挑剔的用户/最苛刻的评审，你会怎么说？**

- "这也太复杂了，能不能简化？"
- "为什么要这么做？有没有更简单的方案？"
- "这个边界情况怎么没考虑？"
- "性能这么差，用户体验会很糟糕吧？"
- "文档太少了，我根本看不懂怎么用。"
- "这个命名太晦涩了，能直观一点吗？"

## 深度检查清单（30 分钟版）

适用于重要任务、生产部署、用户验收前。

### A. 完整功能路径

- [ ] 所有主要功能路径都测试过了吗？
- [ ] 所有次要功能路径都测试过了吗？
- [ ] 所有错误处理路径都测试过了吗？
- [ ] 所有用户角色/权限都测试过了吗？

### B. 长期视角

- [ ] 3 个月后，还能看懂这段代码吗？
- [ ] 如果需求变化，容易修改吗？
- [ ] 如果需要扩展，架构支持吗？
- [ ] 依赖的库/服务会过期/废弃吗？

### C. 成本效益

- [ ] 为了 1% 的场景，投入了 50% 的复杂度值得吗？
- [ ] 有没有过度设计？
- [ ] 有没有 YAGNI (You Aren't Gonna Need It)？

### D. 可观测性

- [ ] 出问题时能快速定位吗？
- [ ] 关键指标有监控吗？
- [ ] 日志足够但不冗余吗？
- [ ] 告警机制完善吗？

## 常见对抗性质疑

| 质疑 | 反问 | 检查点 |
|------|------|--------|
| "太复杂了" | 能不能减少 50% 的代码/步骤？ | 必要性审查 |
| "为什么这样设计" | 有没有更简单的方案？ | 方案对比 |
| "性能问题" | 用户愿意等多久？实际要多久？ | 性能测试 |
| "边界情况" | 如果输入是 [异常值] 会怎样？ | 边界测试 |
| "可维护性" | 3 个月后能看懂吗？ | 代码审查 |
| "安全性" | 最坏情况下会发生什么？ | 安全测试 |

## 验证时机建议

| 任务类型 | 建议检查级别 | 时间预算 |
|---------|------------|---------|
| 小修改/修复 Bug | 快速检查 | 5 分钟 |
| 新功能开发 | 完整检查 | 30 分钟 |
| 重要任务/用户交付 | 深度检查 | 1 小时 |
| 生产部署 | 深度检查 + 人工复核 | 2 小时+ |
