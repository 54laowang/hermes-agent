## 核心对照：Codex vs Claude Code

| 维度 | Claude Code | OpenAI Codex |
|------|-------------|--------------|
| **形态偏好** | 终端/TUI，工程师手感 | IDE 富交互，可视化面板 |
| **战略方向** | 与本地环境共生 | 推向云端/GitHub/Slack/Linear，成为协作工具的一等成员 |
| **上下文管理** | 会话级，易碎 | 三层结构（会话+线程缓存+Memory），沉淀型资产 |
| **用户群体** | 纯工程师 | 想造东西但不想学全部工程仪式的更大人群 |

---

## 上下文工程三层结构

```
第一层：会话上下文 —— 单条线程内累积的信息
第二层：线程缓存   —— 关App不丢，长期保留
第三层：Memory     —— 自动抽取偏好/约定，跨session复用
```

**核心洞察**：终端模式下上下文是易碎品，App形态让它成了沉淀型资产。

---

## Prompting 四大支柱

1. **Goal** —— 你到底要做什么
2. **Context** —— 相关文件/文件夹/文档（**反直觉建议：少用@显式引用**）
3. **Constraints** —— 框架/架构/团队约定（代码看不出来的隐性规则）
4. **Done When** —— 完成判据（**给Agent验证标准，是单点提升质量最有效的杠杆**）

---

## 模型与推理强度分层匹配

```
主线程默认：GPT-5.5 + medium（大多数任务）
子Agent：   GPT-5.4 mini + low/medium（调研、日志解析、跑lint）
硬骨头：    GPT-5.5 + extra high（大代码库、难定位bug、复杂依赖）
```

**核心思想**："用什么级别的算力解什么级别的问题"是新工程素养。

---

## 子 Agent 双重价值

- **并行加速**（常见叙述）
- **保护主线程上下文**（更重要！）—— 避免噪声token污染，主线程只接收结论摘要

**判断标准**：
- 输出量大的任务（日志解析、文件搜索）→ 委派
- 推理量大的任务（代码审查、方案设计）→ 主线程
- 需要用户交互的任务 → 主线程（子Agent无法clarify）

---

## 权限模型的杠杆原理

> **Agent能自主跑的时间越长，你的杠杆越大**

- 事事都要approve = 把Agent降级成"打字快一点的命令补全"
- 细粒度白名单（常用git命令永远放行）
- 分层权限（read-only / write / full access）
- Hooks生命周期（pre-tool-use / post-tool-use）

---

## IDE vs 终端的判断

长期来看 **IDE 形态会赢**：

- 新进场人群结构变了 —— 不是软件工程师，要按钮、要预览、要可视化review
- In-App Browser、图像生成、Computer Use 这些能力接进来后
- 纯TUI是 **能力天花板问题**，不是体验问题

---

## Jevons 悖论与行业判断

> 当一种资源变得更便宜更高效，总需求往往不降反升

AI编码不是消灭工程师岗位，而是把"造东西的能力"扩散到数量级更大的人群。

Codex 正在卡位 **"agentic workspace"** 这个新品类入口。

## 最佳实践

### 线程管理

- 一条线程做一件事，跨任务就开新线程
- 线程过长用 `/compact` 压缩
- 想分叉讨论用 `/fork`
- 本质都是 **保护主上下文窗口**

### 常见错误

- 反复在 prompt 里重复同一规则 → 该沉淀进 `agents.md`
- 不给 Agent 验证标准 → 见 "Done When"
- 跳过 plan mode → 值得做的任务几乎都该先 plan
- 权限设置太严 → 剥夺了 Agent 的"自主跑步时间"
- 多线程改同一文件不开 worktree → 必然撞车
