# 中文小说章节完整修订流程 v3.0

## 三阶段修订流程

```
原始章节 → [阶段零：预检] → [阶段一：结构修订] → [阶段二：语言润色] → 发布
                ↓
           [失败回滚]
```

---

## 阶段零：修订前预检（新增）

### 1. 章节状态检查

**必须确认的三个状态**：

```
□ 文件完整性检查
  - 章节文件是否存在
  - 文件编码是否正确（UTF-8）
  - 文件大小是否异常（过小可能损坏）
  
□ 内容可读性检查
  - 是否有乱码
  - 是否有格式损坏（如截断的段落）
  - 标点符号是否完整
  
□ 关联资源检查
  - 大纲文件是否存在
  - 人物档案是否存在
  - 设定文档是否存在
```

**状态检查失败处理**：
```yaml
文件不存在: 
  → 询问用户文件位置
  → 提供搜索建议
  
文件损坏:
  → 标记损坏位置
  → 询问是否有备份
  → 提供修复方案（如可修复）
  
关联资源缺失:
  → 记录缺失项
  → 询问用户是否继续（可能影响修订质量）
  → 标记"无参考修订"风险
```

### 2. 用户偏好冲突检测（新增）

**偏好冲突类型与处理**：

| 冲突类型 | 检测方法 | 处理机制 |
|---------|---------|---------|
| **风格偏好冲突** | 用户要求风格 vs 原文风格 | 询问优先级：保持原风格 / 按用户要求调整 |
| **人物性格冲突** | 人物档案设定 vs 章节表现 | 标记冲突点，提供两种方案让用户选择 |
| **剧情走向冲突** | 大纲规划 vs 用户修改意见 | 列出差异，询问是否更新大纲 |
| **语言程度冲突** | 用户要求自然度 vs AI润色程度 | 提供润色程度选项：轻度/中度/深度 |

**冲突处理流程**：

```
1. 检测到冲突
   ↓
2. 记录冲突详情（类型 + 具体内容）
   ↓
3. 暂停修订，向用户报告
   ↓
4. 提供解决方案选项
   ↓
5. 等待用户确认
   ↓
6. 按用户选择继续或调整修订策略
```

**冲突报告模板**：

```markdown
## ⚠️ 检测到偏好冲突

**冲突类型**：[类型]
**冲突描述**：
- 原设定：[具体内容]
- 用户要求：[具体内容]

**解决方案**：
A. [方案一说明]
B. [方案二说明]
C. [方案三说明]

请选择处理方案（A/B/C），或提供其他指示。
```

### 3. 修订可行性评估

**评估维度**：

```
修订难度评估：
- 简单：仅语言润色，无逻辑改动
- 中等：部分逻辑调整，人物出场修正
- 困难：时间线重构，设定推翻重写

风险评估：
- 低风险：改动范围小，不影响后续章节
- 中风险：改动中等，需检查后续章节一致性
- 高风险：涉及核心设定，需全面审查

资源需求：
- 是否需要重新阅读相关章节
- 是否需要创建新角色档案
- 是否需要更新大纲
```

## 阶段一：结构修订

### 核心问题排查

1. **逻辑冲突**
   - 时间线矛盾
   - 因果关系断裂
   - 设定前后不一致
   - 人物能力/性格矛盾

2. **人物出场矛盾**
   - 未介绍先出现
   - 重复介绍
   - 角色混淆
   - 对话归属错误

3. **剧情连贯性**
   - 场景转换突兀
   - 信息缺失
   - 节奏问题

### 修订清单

```
□ 时间线一致性检查
□ 人物出场顺序检查
□ 设定前后呼应检查
□ 对话归属正确性检查
□ 场景转换流畅性检查
```

### 结构修订技术

**交叉剪接法**（处理双主线）：
```
主线A推进 → 切镜头到主线B → 切回主线A继续 → 人物自然相遇
```

**时间线重构法**：
```
1. 列出所有事件及时间点
2. 标注人物在场情况
3. 发现冲突 → 调整顺序或增加过渡
4. 验证修改后的合理性
```

## 阶段二：语言润色（Humanizer）

### AI 痕迹识别与去除

基于维基百科"AI 写作特征"识别：

| AI 特征 | 表现 | 修正方案 |
|---------|------|----------|
| 过度结构化 | 层层递进、总分总 | 打散结构、自然过渡 |
| 空洞修饰 | "极其"、"非常"、"令人" | 删除或具体化 |
| 解释性语句 | "这表明"、"可以看出" | 直接陈述 |
| 情感夸张 | "震撼人心"、"深刻反思" | 克制表达 |
| 形容词堆砌 | 多个形容词修饰同一对象 | 精简或意象化 |
| 过渡词滥用 | "然而"、"此外"、"总之" | 减少或替换 |

### 自然化处理

1. **对话自然化**
   - 口语化调整
   - 人物语言风格区分
   - 避免书面语感

2. **叙事流畅化**
   - 消除机械感
   - 增加生活细节
   - 适当留白

3. **节奏优化**
   - 长短句搭配
   - 信息密度控制
   - 情绪起伏设计

## 回滚策略（新增）

### 版本管理机制

**修订前自动备份**：
```
原文件 → 备份文件（.bak）
格式：[原文件名]_[日期时间].bak
示例：chapter01_20260502_082500.bak
```

### 回滚触发条件

| 触发条件 | 回滚级别 | 操作 |
|---------|---------|------|
| 用户不满意修订结果 | 完全回滚 | 恢复.bak文件，重新评估修订策略 |
| 发现新的逻辑冲突 | 部分回滚 | 回退到结构修订阶段，保留预检结果 |
| 偏好冲突未解决 | 暂停修订 | 保存当前进度，等待用户决策 |
| 修订引入新问题 | 阶段回滚 | 回退到上一阶段重新开始 |

### 回滚流程

```
1. 接收回滚信号
   ↓
2. 确认回滚范围（完全/部分/阶段）
   ↓
3. 恢复对应版本
   ↓
4. 记录回滚原因
   ↓
5. 询问用户下一步行动
   ↓
6. 调整修订策略（如需重新修订）
```

### 回滚后处理

```markdown
## 回滚报告

**回滚时间**：[时间]
**回滚原因**：[原因]
**回滚范围**：[完全/部分/阶段]
**已保存进度**：[描述]

**下一步建议**：
- [ ] 调整修订策略
- [ ] 解决偏好冲突
- [ ] 补充参考资料
- [ ] 其他：[用户指定]
```

## 质量标准

### 结构层面

- ✓ 无逻辑冲突
- ✓ 人物出场清晰
- ✓ 剧情连贯流畅

### 语言层面

- ✓ 无明显 AI 痕迹
- ✓ 对话自然生动
- ✓ 叙事节奏舒适

### 风格层面

- ✓ 人物语言有区分度
- ✓ 叙事风格一致
- ✓ 情感表达适度

---

## 质量评分标准（总分50分）

| 维度 | 评分标准 | 满分 |
|------|----------|------|
| **直接性** | 情节推进直接，不绕圈铺垫 | 10 |
| **节奏** | 句子长短交错，段落结尾多样化 | 10 |
| **信任度** | 尊重读者理解能力，不过度解释 | 10 |
| **真实性** | 语言自然，对话符合人物身份 | 10 |
| **精炼度** | 无冗余内容，每段都推动情节 | 10 |

**评分等级**：
- 45-50分：优秀，逻辑通顺，去除AI痕迹
- 35-44分：良好，仍有改进空间
- 低于35分：需要重新修订

## 完整工作流程

### 步骤一：预检阶段（新增）

1. 检查章节文件状态
2. 检查关联资源（大纲、人物档案）
3. 检测用户偏好冲突
4. 评估修订可行性
5. 创建备份文件

### 步骤二：问题定位

- 标记逻辑问题
- 标记 AI 痕迹
- 标记人物/场景问题
- 确认矛盾根源

### 步骤三：结构修正

- 解决逻辑冲突
- 调整人物出场
- 衔接剧情断点
- 验证时间线一致性

### 步骤四：语言润色

- 去除 AI 痕迹
- 自然化表达
- 风格统一

### 步骤五：通读验证

- 检查修订完整性
- 确认自然度
- 最终润色
- 质量评分

### 步骤六：交付与回滚准备

- 输出修订后章节
- 提供修订说明
- 保留备份文件
- 说明回滚方式（如用户需要）

## 实际案例（《时间裂隙》第一章修订）

### 原问题
- 陈默已经被镜子吸进去消失了，林小雨才敲门进来
- 镜子被激活了两次，情节重复
- 造成"两个林小雨"同时存在的逻辑悖论

### 解决方法（交叉剪接）
1. 陈默在库房发现镜子异常，激活启动 → 停在这里
2. 切镜头到时间管理局：警报响起，林小雨接到任务，开车赶往博物馆 → 结束切镜头
3. 切回库房：陈默还粘在镜子前，激活继续，林小雨敲门进来 → 继续推进
4. 镜子爆发，两人同时被卷入 → 情节完整，只有一个林小雨

### 验证
- 从头到尾只有一个林小雨
- 激活只发生一次
- 时间线：陈默启动激活 → 林小雨出发 → 林小雨到达 → 激活完成爆发 → 共同被卷
- 逻辑完全通顺

## 注意事项

### 修订原则
- 用户指出问题后，要完全重构，不要小修小补
- 保留用户原本的核心设定和已经写好的后面章节
- 遵循用户要求，如果用户要求分开展示，就单独输出修改后的章节
- 修改完成后主动检查一遍逻辑链，确保从第一章到当前章节都通顺

### 预检优先
- 遇到文件问题先解决，不要强行修订
- 检测到冲突先报告，不要自行决定
- 高风险修订要提醒用户，获得明确授权

### 回滚准备
- 每次修订前必须创建备份
- 回滚不是失败，是质量保障机制
- 记录回滚原因，优化修订策略
