# Agent 框架架构分析方法论

## 触发条件
当用户：
- 要求分析某个 Agent 框架项目（"分析一下 GenericAgent"、"看看这个项目能学到什么"）
- 分享 GitHub 链接并询问架构设计
- 希望对比不同 Agent 框架的实现差异
- 想从某个 Agent 项目中提取可复用的设计模式

## 标准分析流程

### 第一阶段：项目概览扫描
1. **项目结构快速扫描**
   ```bash
   git clone --depth 1 <repo-url>
   find . -name "*.py" | head -30
   ls -la
   ```
   - 识别入口文件（main.py、agent.py、cli.py）
   - 找到核心模块目录（core、agent、llm、memory、tools）
   - 检查 README 中的架构描述和特性列表

2. **代码规模快速评估**
   - 文件数量、总行数、主要语言分布
   - 依赖项检查（requirements.txt、pyproject.toml）

---

### 第二阶段：核心架构深层分析

#### 🔄 **Agent Loop 核心循环**（最高优先级）
这是所有 Agent 框架的灵魂，必须找到并彻底理解：

```
[User Input] → [System Prompt + Context] → [LLM Call] → [Tool Dispatch]
     ↑                                                         ↓
     └──────────────── 循环或终止决策 ←─────────────────────────┘
```

**关键发现点：**
- 循环入口文件和函数名
- 最大轮次限制（max_turns）
- 工具调用解析机制（JSON 解析？正则？function call？）
- 终止条件判断逻辑（如何判断任务完成？）
- 无工具调用时的特殊处理（fallback 机制）

#### 🧠 **记忆系统分层设计**
分析 5 层记忆模型的具体实现：

| 层级 | 检查要点 |
|-----|---------|
| **L0 系统提示** | 静态还是动态生成？包含哪些行为约束？ |
| **L1 洞察索引** | 技能目录如何组织？SOP 如何索引？ |
| **L2 全局事实** | 用户偏好、环境配置、凭证如何持久化？ |
| **L3 技能/SOP** | SOP 是代码还是 Markdown？如何被读取和执行？ |
| **L4 会话归档** | 完整会话记录在哪里？如何被复盘和学习？ |

**记忆更新机制：**
- 任务进行中的临时记忆（working checkpoint）
- 任务完成后的经验沉淀（post-task SOP generation）
- 跨任务的上下文保持策略

#### 🛠️ **原子工具集设计**
分析核心工具的实现模式：

| 工具类型 | 检查要点 |
|---------|---------|
| **代码执行** | 沙箱隔离？超时机制？流式输出？头注入？ |
| **文件读写** | 截断保护？diff 验证？唯一匹配校验？ |
| **浏览器控制** | CDP 集成？登录态保持？反自动化绕过？ |
| **人类反馈** | 如何中断循环等待用户输入？ |

**关键设计模式：**
- `BaseHandler` 基类 + `dispatch` 动态分发
- 前置钩子（pre-hook）和后置钩子（post-hook）
- 生成器模式实现流式输出

#### 📋 **计划模式与验证机制**
- Plan 文件格式（Markdown Checklist？结构化数据？）
- 「任务完成声明」的拦截机制（幻觉防护）
- VERIFY 验证步骤的强制执行
- 子代理验证的调用流程

### 第三阶段：关键工程实践提取

#### ✅ **Token 优化策略**
- 工具描述的懒加载/按需重置
- 输出智能截断算法
- 上下文压缩技术
- 代码块预览压缩

#### ✅ **前端后端解耦设计**
- 多平台接入层（Web、Telegram、微信、飞书、钉钉...）
- 统一的 Agent 内核接口
- 消息队列/任务队列设计

#### ✅ **多模型适配层**
- 模型抽象接口设计
- 不同 provider 的差异抹平
- Mixin/混合模型策略
- 模型切换时的状态迁移

## ⚠️ 常见陷阱提醒

1. **不要只看 README** - 很多高级特性文档不会写，必须读源码
2. **不要忽略注释掉的代码** - 往往记录了作者的探索历程和放弃的原因
3. **关注 guard clauses** - 边界条件处理体现了作者踩过多少坑
4. **找到 `try/except` 块** - 异常处理揭示了系统的脆弱点
5. **搜索 `HACK`、`FIXME`、`TODO`** - 这些是作者对自己的诚实评价
