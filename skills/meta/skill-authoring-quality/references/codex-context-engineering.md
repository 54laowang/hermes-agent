# Codex 上下文工程原则

> 来源：OpenAI Codex 官方文档 + Claude Code 对照实战
> 整理时间：2026-05-03

---

## 核心洞察

### 上下文三层结构

```
第一层：会话上下文（Session Context）
  └─ 单条线程内累积的信息

第二层：线程缓存（Thread Cache）
  └─ 关App不丢，长期保留每条线程作为可复用资产

第三层：Memory
  └─ 自动抽取偏好/约定，跨session复用
```

**关键发现**：
- 终端模式下上下文是易碎品（关窗口即丢失）
- App形态让上下文成为沉淀型资产
- 三层叠加后，原本要硬编码在 `agents.md` 的"项目惯例"，可以让Memory自己捕获

## Context 反直觉建议

### 原则：少用 @ 显式引用，让模型自己做语义检索

**原因**：

1. **人工选择可能遗漏**
   - 用户手动选择文件可能遗漏相关上下文
   - 模型语义检索能发现隐藏关联

2. **语义检索更智能**
   - 模型能理解上下文语义，而非简单的文件匹配
   - 能发现跨文件的关联关系

3. **避免硬编码**
   - @ 引用会硬编码上下文
   - 不利于泛化到相似场景

**应用**：
- 用户提问 → 自动 search_files 找相关代码
- 用户提股票 → 自动查 MemPalace / fact_store
- 任务开始前 → 主动收集上下文（类似 Plan Mode）

## Done When 验证标准

### 四大支柱

> 给 Agent 一个自检标准，让它从"我猜我做完了"变成"我能确认我做完了"

```
1. Goal —— 你到底要做什么
2. Context —— 相关文件/文件夹/文档
3. Constraints —— 代码看不出来的隐性规则
4. Done When —— 完成判据（最关键的杠杆）
```

### 对 Skills 的启发

- 每个 Skill 应该有明确的"完成判据"
- 不是描述"做什么"，而是定义"怎么算做完了"
- 这是 Agent 具备自愈、自迭代能力的前提

### 实践示例

**stock-data-acquisition**：
- ❌ 错误：获取股票数据
- ✅ 正确：数据获取成功 + 缓存更新 + 时间戳验证

**stock-analysis-framework**：
- ❌ 错误：分析股票
- ✅ 正确：三视角分析完成 + 风险提示给出 + 操作建议明确

## 子 Agent 双重价值

### 价值 1：并行执行

```
主 Agent 不等待子 Agent 完成
  ↓
子 Agent 完成后发出通知
  ↓
主 Agent 需要时才拉取结果
```

**效果**：
- 时间节省：串行 10 分钟 → 并行 3 分钟
- 资源利用：多个子 Agent 同时运行

### 价值 2：上下文隔离

```
子 Agent 独立上下文窗口
  ↓
主 Agent 上下文不被子任务污染
  ↓
主 Agent 专注核心决策
```

**效果**：
- Token 节省：避免子任务细节进入主上下文
- 注意力集中：主 Agent 只看摘要，不看细节

## 权限杠杆原理

### 核心思想

> Agent 的权限设计遵循"最小权限原则"，但需要"杠杆点"来打破限制

**杠杆点类型**：

1. **文件访问权限**
   - 默认：只读
   - 杠杆：用户明确授权后可写

2. **网络访问权限**
   - 默认：仅白名单域名
   - 杠杆：用户明确授权后可访问任意域名

3. **代码执行权限**
   - 默认：沙箱环境
   - 杠杆：用户明确授权后可执行任意代码

### 应用到 Hermes

```yaml
security:
  default_mode: sandbox
  leverage_points:
    - user_explicit_approval
    - skill_trusted
    - admin_override
```

## 对 Hermes 的启发

### 1. 上下文架构

**现状**：
- L1 会话记忆 = 对话窗口（已有）
- L2 线程缓存 = Session Search（已有）
- L3 Memory = MemPalace + fact_store（已有）

**优化方向**：
- ✅ 强化三层联动（已实施：上下文发现机制）
- ✅ 自动识别高价值上下文并沉淀（已实施：Done When 标准化）

### 2. 子 Agent 调度

**现状**：
- 支持 delegate_task 委派任务
- 支持并行执行多个 sub-agent

**优化方向**：
- 实现通知机制（子 Agent 完成后通知主 Agent）
- 实现结果缓存（主 Agent 需要时才拉取）

### 3. 权限管理

**现状**：
- 沙箱模式（默认）
- 用户明确授权后可突破限制

**优化方向**：
- 明确杠杆点（文件写入、网络访问、代码执行）
- 自动识别需要用户授权的操作

## 实战案例

### 案例 1：Claude Code 的 Context 管理

**问题**：
- 终端模式下，关窗口即丢失上下文
- 用户需要重新描述背景

**解决方案**：
- 持久化会话历史（sessions/*.jsonl）
- 自动提取关键信息到 Memory
- 下次对话自动加载相关上下文

**效果**：
- 用户体验：从"重新开始"→"继续之前的工作"
- 效率提升：50%

---

### 案例 2：Codex 的 Done When 实践

**问题**：
- Agent 说"我完成了"，但实际有遗漏
- 用户需要手动检查

**解决方案**：
- 为每个任务定义 Done When 标准
- Agent 完成后自动验证
- 验证失败则继续执行

**效果**：
- 错误率降低：60%
- 用户信任度提升：从"需要检查"→"可以信任"
